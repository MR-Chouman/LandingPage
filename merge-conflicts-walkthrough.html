Merge Conflicts Walkthrough

This exercise is designed to help both teammates practice merging branches and resolving conflicts in a realistic way. Your goal is not to guess which side is “correct”, but to understand what each branch changed and produce one clean final result.

Workflow Goal

The aim of this version of the exercise is for both students to experience merge conflicts, not only the person who merges second into main.

To do that, each student will merge the other student's branch into their own working branch and resolve conflicts there.

Starting Point

Both students should start from the same clean base on main:

git checkout main
git pull

Then each student checks out their assigned branch:

  • Student A works on conflicts-nature
  • Student B works on conflicts-desert

Commands To Run

Student A:

git checkout conflicts-nature
git merge conflicts-desert

Student B:

git checkout conflicts-desert
git merge conflicts-nature

Because both branches touch the same hotspots in different ways, both students should now experience merge conflicts on their own branch.

What To Do When Git Stops

  1. Run git status to see which files are in conflict.
  2. Open each conflicted file in your editor.
  3. Look for Git markers: 
    <<<<<<< HEAD
your current version
=======
incoming branch version
>>>>>>> conflicts-desert
  4. Read both sides carefully before editing.
  5. Decide what the final version should be.
  6. Remove the Git markers after resolving the block.
  7. Repeat until the file is fully clean.

Why This Workflow Matters

  • Both students must read and understand the conflicting code.
  • Both students must decide what a clean final version should look like.
  • Both students practice the same problem-solving skill instead of only one student seeing the conflict.
  • You can compare how each student resolved the same problem and discuss their reasoning afterward.

How To Think About The Conflicts

  • Treat conflicts as two valid attempts that now need to be reconciled.
  • Do not resolve them line by line without understanding the surrounding code.
  • Read the full function or section around the conflict before deciding.
  • Watch for duplicate helpers, duplicate event listeners, or repeated logic.
  • If both sides do the same thing in different styles, choose one consistent pattern.
  • If you combine both sides, make sure the result is still clean and non-redundant.

Minor Vs Major Conflicts

Minor conflicts are simpler and usually involve comments or small structural differences.

Major conflicts require more care because they involve larger blocks such as helper functions, toggles, or update logic.

For the bigger conflicts, focus on intent:

  • What is this block responsible for?
  • How is each branch trying to structure that responsibility?
  • What final version would be easiest for a teammate to read and maintain?

Finishing The Merge

After all conflicts are resolved:

git add nature/index.html nature/script.js desert/index.html desert/script.js
git commit

At this point, each student has a branch that contains their own work plus a resolved merge of the other branch.

After The Conflict Exercise

  • Compare the two final resolutions side by side.
  • Discuss whether both students kept the code coherent and free of duplication.
  • Review whether the final result is easy for another teammate to read.
  • If needed, choose one final version to merge into main.

Useful Checks Before Committing

git status
rg '<<<<<<<|=======|>>>>>>>'

The search command helps confirm that no conflict markers are left behind.

Final Sanity Check

  • Open both pages in the browser after the merge.
  • Make sure the nature page still loads and works.
  • Make sure the desert page still loads and works.
  • Make sure the JavaScript still runs without obvious errors.

Mindset

Good conflict resolution is not about speed only. It is about understanding intent, keeping the result coherent, and leaving the codebase in a state that another teammate can follow easily.
